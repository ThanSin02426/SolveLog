# Promotion and small-revenue plan

## Positioning

Do not market SolveLog as another automatic uploader. Lead with the trust and revision benefits:

> A private, topic-first LeetCode archive that never asks for access to all your repositories.

The strongest differentiators are:

1. one-repository permission;
2. zero-access download mode;
3. domain → difficulty organisation;
4. preserved personal notes;
5. no problem-statement copying;
6. open, inspectable extension core.

## Launch sequence

### Phase 1: prove reliability

- Publish the source repository with a clear security model.
- Recruit 20–40 testers from college coding groups and developer communities.
- Track only voluntary feedback; do not add hidden analytics.
- Fix LeetCode UI breakages quickly.
- Ask satisfied users for honest store reviews after several successful saves, not immediately after installation.

### Phase 2: public launch

- Chrome Web Store listing with security-first screenshots.
- A 45-second demo showing Accepted → GitHub commit → topic folder.
- Launch posts for LinkedIn, X, Reddit communities, Dev.to and relevant Discord groups.
- Share a technical post explaining why broad GitHub OAuth scopes are risky and how repository-scoped access works.
- Create short videos around practical searches: “organise LeetCode by topic”, “sync LeetCode safely”, and “build a DSA revision repo”.

### Phase 3: compounding distribution

- Add a tasteful “Generated by SolveLog” line only to the root README, with an opt-out before public launch.
- Provide GitHub profile README badges showing total solved by difficulty.
- Add weekly revision lists and streak summaries.
- Partner with DSA instructors, coding clubs and placement-preparation creators.
- Publish migration tools from common flat LeetCode repositories.

## What to keep free

The core automatic save, one repository, all languages, topic organisation, manual notes and download mode should stay free. Charging for the basic sync will make it difficult to compete with established free extensions.

## Sensible paid features

A small optional Pro plan can fund maintenance:

- encrypted cloud backup across devices;
- revision reminders and spaced repetition;
- advanced dashboards and custom topic rules;
- multiple target repositories;
- interview-sheet exports;
- team or classroom dashboards;
- optional AI-generated explanation drafts, clearly marked for review.

Suggested starting experiment for India:

- Free: core extension
- Supporter: ₹99–₹149 one-time, badge and early features
- Pro: ₹49–₹99/month or ₹499–₹799/year after the product has recurring value
- Student lifetime launch offer: ₹299–₹499, capped to early adopters

Do not charge until the extension survives several weeks of real LeetCode usage without losing submissions.

## Payment architecture

The Chrome Web Store should distribute the extension; billing should be handled by your own HTTPS application using a payment provider. Keep payment secrets on the server. The extension should receive only a signed entitlement or short-lived session token.

For Indian users, compare Razorpay and Stripe for your exact business/entity eligibility and current fees before implementation. Do not embed provider secret keys in the extension.

## Low-risk revenue first

The easiest initial income is voluntary support rather than subscriptions:

- GitHub Sponsors or Buy Me a Coffee;
- a one-time supporter licence;
- sponsored placement in your own launch content, not inside users' coding pages;
- paid setup for colleges or coding cohorts.

Avoid ads. Chrome policy limits using extension data for personalised advertising, and ads would weaken the privacy-first positioning.

## Metrics that matter

Measure only with consent and preferably from store/dashboard aggregates:

- installation-to-first-save rate;
- seven-day active retention;
- successful saves versus retries;
- uninstall rate after onboarding;
- review score and support response time;
- free-to-supporter conversion.

Do not collect source code, repository names or browsing behaviour for analytics.
